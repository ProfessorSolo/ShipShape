It is crucial to understand that an application doesn't just "move" when it gets deployed. Its entire surrounding context changes.

In Lesson 03, we solved the local development problem. We used Docker Compose to create a nice, tidy local network. We defined an app container and a database container. They sat side-by-side, entirely under our control.

Deployment tears that arrangement apart.

## The Architectural Shift

When we transition to hosted delivery, the monolithic local setup gets unbundled into independent, specialized services. 

Here is exactly what changes:

- **The App is Hosted Elsewhere:** The application no longer runs on our laptop's hardware. It runs on a managed platform provider.
- **The Database Leaves the Sibling Network:** We do not host the database next to the app in the same container cluster. The application data will live in a specialized, highly available database engine far away from the application code.
- **Configuration Migrates:** We can no longer rely on a local `.env` file sitting comfortably on our hard drive. Variables and secrets must be injected dynamically by the host environment.
- **The Workflow Matures:** We stop using `docker-compose up` to run the latest code. Deployment is now a workflow step—triggered directly from a version-controlled repository (Git).

<CautionBox title="Broken Assumptions">
  Every shortcut we took during local development is violently exposed during deployment. Hardcoded ports fail. Localhost URLs break. Hardcoded database connections shatter. We cannot hide infrastructure assumptions in a public deployment.
</CautionBox>

<NoteBox title="Think Like a Operator">
  Moving from local to live means treating the application as a stateless delivery mechanism. It should expect to be started, stopped, configured, and routed entirely by external forces.
</NoteBox>

This modular approach is what allows modern applications to scale and survive crashes without taking their databases down with them. 

The first piece of this new decoupled architecture is finding a proper harbor for the application code to actually run. Let's head over to the host.
