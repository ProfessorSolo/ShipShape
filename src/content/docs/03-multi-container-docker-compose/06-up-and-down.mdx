We have our `compose.yaml` blueprint ready. In our manual labs, this is the point where we would be typing out six or seven different commands just to get the network and containers started. With Compose, we reduce that entire operational headache to a single command.

## Ignition: `docker compose up`

This is the most important command in your toolkit. When you run this, Compose reads your manifest, builds your custom images, pulls your dependencies, creates the network, and launches every service in the correct order.

```bash
docker compose up
```

### The Detached Flag (`-d`)

By default, running `up` will lock your terminal and stream the logs from every container into one messy window. While great for a quick look, it isn't practical for development. We almost always use **Detached Mode**:

```bash
docker compose up -d
```

This tells Compose to "fire and forget"—the containers will start and run in the background, leaving your terminal open for other work.

## Teardown: `docker compose down`

When we are finished with our work, we don't want orphan containers or unused network bridges hogging our system resources.

```bash
docker compose down
```

Unlike `docker stop`, which just pauses the process, `down` is a total teardown. It stops the containers, removes them, and deletes the virtual bridge network. Because our configuration is **declarative**, we aren't afraid of this destruction—we can bring the exact same environment back to life in seconds with another `up` command.

<NoteBox title="Monitoring the Pulse">
  Even in detached mode, we need to see what our services are doing. We can
  check the status of our fleet at any time: * **`docker compose ps`**: Shows
  which containers are running and what ports they are using. * **`docker
  compose logs -f`**: Shows a real-time, unified log stream from all services.
</NoteBox>

<CautionBox title="Stop vs. Down">
  * **`docker compose stop`**: Freezes the containers. They still exist, but
  they aren't using CPU. * **`docker compose down`**: Erases the containers. In
  this course, we prefer `down` to ensure we are always testing against a fresh,
  deterministic state.
</CautionBox>
