Our database is idling in the background, but it's currently an island. We need to build our **API Service**—the logic tier that will bridge the gap between our raw data and our end users.

For this project, we are building a minimal Node.js application using the **Express** framework and **Mongoose** to talk to MongoDB.

## 1. The Application Files

Create a new directory for your API and initialize these three files.

### File A: `package.json`

This defines our dependencies. We need `express` for the server and `mongoose` for the database connection.

```json
{
  "name": "shipshape-api",
  "version": "1.0.0",
  "main": "server.js",
  "dependencies": {
    "express": "^4.18.2",
    "mongoose": "^7.5.0"
  }
}
```

### File B: `server.js`

The logic for our API. Note the `MONGO_URI`—it is currently pointing to `localhost`.

```javascript
const express = require("express");
const mongoose = require("mongoose");
const app = express();

const PORT = 3000;
const MONGO_URI = "mongodb://localhost:27017/shipshape";

mongoose
  .connect(MONGO_URI)
  .then(() => console.log("Connected to the ShipShape Database!"))
  .catch((err) => console.error("Connection Error:", err));

const InventorySchema = new mongoose.Schema({
  item: String,
  status: String,
  quantity: Number,
});

const Inventory = mongoose.model("Inventory", InventorySchema, "inventory");

app.get("/api/inventory", async (req, res) => {
  try {
    const items = await Inventory.find();
    res.json(items);
  } catch (error) {
    res.status(500).json({ error: "Failed to fetch inventory" });
  }
});

app.listen(PORT, () => {
  console.log(`API Service listening on port ${PORT}`);
});
```

### File C: `Dockerfile`

The blueprint that tells Docker how to build our vessel.

```dockerfile
FROM node:20-slim
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

## 2. Building and Launching

Now, we transform these files into a living container. Run these commands in your terminal:

```bash
# 1. Build the API image
docker build -t shipshape-api .

# 2. Launch the API container
docker run -d -p 3000:3000 --name api_service shipshape-api
```

## The "Silent" Failure

If we check our container logs right now, we’ll see a major problem:

```bash
docker logs api_service
```

Instead of a "Connected" message, you’ll likely see a massive connection timeout or an `ECONNREFUSED` error.

<CautionBox title="The Localhost Trap">
  Even though our `db_service` is running, our API is screaming into the void.
  It is looking for MongoDB at `localhost`, but inside a container, `localhost`
  means **itself**. Our API is trying to find a database inside its own walls,
  unaware that the real data is sitting in a separate container right next to
  it.
</CautionBox>
