We’ve discovered the **"Localhost Trap."** Our API is looking for a database inside its own walls, while our database is sitting in a separate, isolated container. To make them talk, we have to manually provision a private bridge and re-launch our entire fleet.



## 1. Creating the Virtual Bridge

By default, containers are isolated. We need to create a dedicated communication channel so our services can "see" each other by name. Run the following command:

```bash
docker network create shipshape-network
```

## 2. Re-Launching the Fleet

A container cannot join a new network while it is already running. We have to "dry dock" our current containers and re-launch them with specific networking instructions.

```bash
# 1. Remove the existing isolated containers
docker rm -f db_service api_service

# 2. Launch the Database on the new network
docker run -d --name db_service --network shipshape-network mongo:7.0

# 3. Launch the API on the new network
docker run -d -p 3000:3000 --name api_service --network shipshape-network shipshape-api
```

## 3. The Final Hurdle: Updating the Code

Even though they are on the same network, our `server.js` is still hardcoded to look for `localhost`. On a Docker network, we can use the **Container Name** as a hostname. 

You must now go back into your `server.js` and update the connection string:

```javascript
// Change this:
const MONGO_URI = 'mongodb://localhost:27017/shipshape';

// To this:
const MONGO_URI = 'mongodb://db_service:27017/shipshape';
```

**Because you changed the source code, you must rebuild the image and restart the container again:**

```bash
docker build -t shipshape-api .
docker rm -f api_service
docker run -d -p 3000:3000 --name api_service --network shipshape-network shipshape-api
```

## The Manual Labor Critique

If that felt like a lot of typing, that’s because it was. To get two simple services talking, we had to:
1.  **Manually** create a network.
2.  **Manually** track container names.
3.  **Manually** stop and remove containers.
4.  **Manually** rebuild images for every configuration change.
5.  **Manually** remember the exact `docker run` flags for ports, names, and networks.

<CautionBox title="Brittle Infrastructure">
  Imagine doing this for a stack with 10 services. If you forget a single `--network` flag or a single port mapping, the entire system collapses. This manual workflow is **imperative**—you are telling Docker *how* to do every tiny step, and if you trip, the whole fleet sinks.
</CautionBox>

<NoteBox title="Success!">
  If you check your logs now (`docker logs api_service`), you should finally see: **"Connected to the ShipShape Database!"** You can now visit `http://localhost:3000/api/inventory` in your browser and see your seeded data.
</NoteBox>
