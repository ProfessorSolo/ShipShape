If the manual labor of the last section felt unsustainable, you’re right—it is. Professional engineers don't sit in terminal windows for forty minutes trying to remember which network flag goes with which container name.

Instead, we use **Docker Compose**. 

Compose allows us to define our entire multi-container fleet in a single, human-readable text file called a **Manifest**. Instead of telling Docker *how* to build every bridge and launch every ship, we simply tell it *what* our ideal harbor should look like.



## The Master Manifest: `compose.yaml`

In the root of your project directory (alongside your `Dockerfile`), create a new file named `compose.yaml`. This file will replace every single manual command we ran in the previous lesson.

```yaml
services:
  api_service:
    build: .
    ports:
      - "3000:3000"

  db_service:
    image: mongo:7.0
```

### What happened to the Network?
You’ll notice there is no `networks:` block in this file. One of the primary benefits of Compose is that it **automatically** creates a private bridge network for your project and attaches every service to it. The containers can already "see" each other using their service names as hostnames.

## Imperative vs. Declarative

This shift represents a fundamental change in your engineering philosophy:

* **Imperative (The Old Way):** You issue a sequence of commands to change the state of the system step-by-step. If one step fails, the whole process breaks.
* **Declarative (The Compose Way):** You define the **desired end-state**. You hand this file to Docker Compose and say, "Make this real." Compose then calculates the most efficient way to bring your vision to life.

<NoteBox title="One File to Rule Them All">
  By committing this single YAML file to your Git repository, you ensure that every other developer on your team can launch the exact same environment with the exact same settings by running one single command. No more "It worked on my machine" excuses.
</NoteBox>

<CautionBox title="The Hyphen History">
  You may see older tutorials using a file named `docker-compose.yml` (with a hyphen and a .yml extension). While the Docker engine still recognizes that format, the modern professional standard is the cleaner, hyphen-less `compose.yaml`. 
</CautionBox>
