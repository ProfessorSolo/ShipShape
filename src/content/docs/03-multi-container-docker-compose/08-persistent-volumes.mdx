In the world of Docker, containers are meant to be **ephemeral**. They are designed to be stopped, deleted, and replaced without a second thought. However, your data is sacred. 

If we store our MongoDB records inside the container’s internal filesystem, those records are deleted the moment we run `docker compose down`. To solve this, we must "drop an anchor" by using **Named Volumes**.



## 1. Stateless vs. Stateful

As engineers, we categorize our services into two types:
* **Stateless (The API):** If this container is deleted, we don't care. We have the source code and the Dockerfile to rebuild it exactly as it was.
* **Stateful (The Database):** If this container is deleted, we lose our "source of truth." We need a way to store this data **outside** the container.

## 2. Implementing the Anchor

We need to update our `compose.yaml` to include a volume. This requires two steps: declaring the volume at the top level and "mounting" it to the service.

```yaml
services:
  api_service:
    build: .
    ports:
      - "${PORT}:${PORT}"
    environment:
      - DB_USER=${MONGO_USER}
      - DB_PASS=${MONGO_PASS}
    depends_on:
      - db_service

  db_service:
    image: mongo:7.0
    environment:
      - MONGO_INITDB_ROOT_USERNAME=${MONGO_USER}
      - MONGO_INITDB_ROOT_PASSWORD=${MONGO_PASS}
    volumes:
      - shipshape_data:/data/db

volumes:
  shipshape_data:
```

### Breaking Down the Volume Logic:
1.  **The Global Declaration:** At the very bottom, we added `volumes: shipshape_data:`. This tells Docker to reserve a permanent, named storage space on your physical hard drive.
2.  **The Service Mount:** Under `db_service`, we added `- shipshape_data:/data/db`. This "plugs" our permanent storage into the exact folder where MongoDB stores its data internally.

## 3. Testing the Persistence

1.  Run `docker compose up -d`.
2.  Add some data to your database (using the seed command from Page 01).
3.  Run `docker compose down`. (The containers are now gone).
4.  Run `docker compose up -d` again.
5.  Check your data: `docker exec -it db_service mongosh shipshape --eval "db.inventory.find()"`.

**Your data is still there.** The container was new, but the volume was old.

<NoteBox title="Why /data/db?">
  Every official image has a specific folder where it stores its "state." For MongoDB, that folder is `/data/db`. For Postgres, it’s `/var/lib/postgresql/data`. You must always check the official documentation on Docker Hub to find the correct internal path to anchor.
</NoteBox>

<CautionBox title="The -v Danger Zone">
  If you truly want to wipe your data and start from scratch, you can run `docker compose down -v`. The `-v` flag tells Docker to destroy the containers **and** the volumes. Use this only when you are absolutely certain you want to delete your records forever.
</CautionBox>
