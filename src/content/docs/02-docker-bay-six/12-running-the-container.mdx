## Start the App from the Image

We have a built image named `my-simple-server`. To create and run a live container from that image, we run:

```bash
docker run -p 3000:3000 my-simple-server
```

### Breakdown of the command

- `docker run`: creates and starts a new container based on an image.
- `-p 3000:3000`: maps host port `3000` to container port `3000`.
- `my-simple-server`: the name (tag) of the image we want to run.

## Port Mapping Details

Port mapping is one of the most common early confusion points for Docker students.

{/* T.A. Watts Note: A clean port mapping diagram showing traffic flowing from Localhost to the Host Port, bridging across to the Container Port, and finally hitting the Node app inside would be invaluable here. */}

The format is always:

```text
hostPort:containerPort
```

So if we ran:

```bash
docker run -p 8080:3000 my-node-app
```

That means:

- The host machine (your laptop) listens on port `8080`.
- Traffic hitting `8080` is forwarded to port `3000` _inside_ the container, which is where your Node app is listening.

You would visit the app at `http://localhost:8080`.

<CautionBox title="Port Collisions">
  If another application on your laptop is already using port 3000, `docker run
  -p 3000:3000` will fail. You'll need to map it to an open host port, like `-p
  3001:3000`. Note that the container port usually stays whatever the app was
  programmed to listen on.
</CautionBox>

## Test It Out

If the Node app is listening on the expected port and starts cleanly, it should now be fully reachable in your browser.

<NoteBox title="A Successful Voyage">
  That is the big moment: your exact same application is now running cleanly
  inside an isolated container, totally divorced from whatever weird global
  tools might be installed on your laptop.
</NoteBox>
